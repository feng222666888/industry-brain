# 行业大脑三层架构 - 前端页面完整设计方案

## 0. 现状诊断

当前 6 个页面（cockpit / device / optimize / catalyst / evolution / agent-ops）存在以下问题：

- **架构不可见**：底座支撑层（知识中心、数据治理、工具库、模型网关）在前端完全缺失
- **模块错位**：agent-ops 页面混合了 Agent 注册（属核心运行）、数据来源健康度（属底座支撑）、审计日志（属底座支撑）
- **粒度不足**：evolution 页面同时展示在线/离线两条流水线，无法分别深入
- **层次缺失**：侧边栏无法体现三层架构的从属关系

## 1. 侧边栏结构（精确映射 architecture.md 三层 + 应用层）

```
企业驾驶舱               ← 应用层（行业场景入口）
  总驾驶舱         /cockpit
  设备预测维护     /device
  工艺参数寻优     /optimize
  催化剂研发       /catalyst

底座支撑                 ← 第一层（基础设施与知识）
  知识中心         /knowledge           ← NEW
  数据治理         /governance           ← NEW（吸收 agent-ops 部分内容）
  工具库           /tools                ← NEW
  模型网关         /model-gateway        ← NEW

核心运行                 ← 第二层（Agent 运行时）
  记忆系统         /memory               ← NEW
  Agent 生产系统   /agent-factory        ← NEW（吸收 agent-ops 部分内容）
  Multi-Agent 引擎 /multi-agent          ← NEW
  可观测系统       /observability        ← NEW（吸收 agent-ops 部分内容）

自我进化                 ← 第三层（持续进化）
  在线演进         /evolution/online     ← NEW（拆分自 evolution）
  离线进化         /evolution/offline    ← NEW（拆分自 evolution）
```

旧页面处置：`/evolution` 拆为 online + offline；`/agent-ops` 拆散到 governance / agent-factory / observability，路由废弃。

## 6. 自检与风险

**Q: 10 个新页面是否过多导致单页内容过薄？**
每个页面至少对应 3-4 个内容区域。即使部分数据为 demo/mock，页面结构完整即可证明架构模块已实现。POC 阶段，"看得见架构" 比 "每页数据真实" 更重要。

**Q: 工具库和模型网关是否内容重复？**
不重复。工具库展示 Agent 可调用的"能力"（检索/仿真/浏览器/API），模型网关展示"推理引擎"（LLM 模型配置和状态）。前者是手，后者是脑。

**Q: 旧路由(/evolution, /agent-ops)如何处理？**
保留路由但做 302 重定向：`/evolution` → `/evolution/offline`，`/agent-ops` → `/agent-factory`。确保外部链接不断裂。

**Q: 28 个新 API 是否工程量过大？**
多数 API 是对已有类（AgentRegistry / SessionMemory / SessionTracer / EvolutionService）的 HTTP 封装，平均每个端点 10-20 行代码。7 个 router 可并行开发。
