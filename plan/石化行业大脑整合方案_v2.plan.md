# 石化行业大脑产品整合方案（精简版）

## 1. 目标与边界

### 1.1 产品目标

- 面向头部石化企业，交付可演示、可部署、可扩展的行业大脑产品。
- 以三大场景作为能力切片：设备维护（核心运行）、工艺寻优（进化引擎）、催化剂研发（底座支撑）。
- 形成差异化能力：在线演进+离线进化双轮驱动，持续沉淀策略资产。

### 1.2 客户优先级

- 当前主线：企业客户（如山东京博石化）。
- 远期扩展：政府监管视图（基于企业真实运营聚合数据，而非Mock）。

### 1.3 里程碑（相对阶段）


| 里程碑 | 目标          | 关键交付            |
| --- | ----------- | --------------- |
| V1  | 场景三可完整演示    | Agent引擎+设备维护驾驶舱 |
| POC | 至少2家头部企业可演示 | 三场景企业版工作台+POC报告 |
| 规模化 | 可复制交付       | 私有化部署包+行业扩展机制   |
| 远期  | 政府版启动       | 政府视图BFF与聚合指标体系  |


---

## 2. 架构与扩展策略

### 2.1 三层架构

- 核心运行：记忆系统、Agent生产系统、Multi-Agent协同、可观测系统。
- 底座支撑：知识中心（图谱+文档+向量检索）、工具库（检索/仿真/优化）。
- 自我进化：在线演进（实时信号驱动）+离线进化（批量回放与策略变异）。

### 2.2 扩展原则

- 新场景优先走“配置扩展”而非“引擎改造”。
- 新行业通过 `industry_id` 做逻辑隔离（API、知识图空间、策略资产）。
- 数据与知识统一走治理流水线，禁止绕过治理直接入库。

---

## 3. 三大场景（企业版）

### 3.1 优先级与定位

- 优先级：场景三 > 场景一 > 场景二（按数据就绪度、技术风险、客户共鸣综合排序）。

### 3.2 场景摘要

1. 场景三：关键设备预测性维护
  - 目标：告警-诊断-维修SOP闭环，降低非计划停车风险。  
  - 指标：预测准确率、SOP完整率、端到端延迟。
2. 场景一：炼化工艺参数在线寻优
  - 目标：收率提升、能耗下降、策略可追溯。  
  - 指标：收率提升%、能耗下降%、策略收敛速度。
3. 场景二：催化剂电镜图像智能识别
  - 目标：图像特征识别+图谱推理+研发建议追溯。  
  - 指标：识别准确率、推理可解释性、文献追溯完整性。

## 4. 数据策略（生产可用升级）

### 4.1 结论

本方案采用 **“一次性冷启动 + 周期性增量 + 事件触发补采”** 的混合策略，不采用一次性静态抓取。

### 4.2 三态抓取策略

1. 冷启动全量：首次构建知识与特征基线（用于首版上线与基准评测）。
2. 周期增量：按来源级别设置小时/日/周更新频率。
3. 事件触发：法规更新、事故通报、热点文献等触发即时补采。

### 4.3 白名单与合规

- 仅允许白名单来源进入采集调度。
- 每条数据保留可追溯元数据：`source_url`、`acquired_at`、`industry_id`、`scenario_id`、`content_hash`。
- robots/copyright/审计日志为硬门禁。

### 4.4 去重与增量

- URL标准化 + 内容指纹双去重。
- 基于 `last_seen_at` + `content_hash` 做增量更新，仅处理新增/变更内容。

## 5. 数据治理与进化引擎闭环

### 5.1 治理框架（四维）

- 质量治理：完整性、时效性、一致性、唯一性、物理边界。
- 语义治理：术语标准化与跨源映射。
- 合规治理：来源合法性与访问审计。
- 资产沉淀：知识库、特征库、策略训练集、评测基准集。

### 5.2 质量分数接入策略准入（必须代码化）

- 输出统一 `data_quality_score`（0~1）用于策略准入与降权。
- 准入规则：
  - 高质量：正常进入策略评分；
  - 中质量：降权进入；
  - 低质量：阻断执行或仅建议态。
- 评分融合建议：
  - `final_score = model_score * quality_weight * safety_weight`

### 5.3 漂移与再进化

- 检测数据分布/语义/来源漂移。
- 漂移超阈值触发离线再训练与再进化任务。
- 全程写审计日志，支持回放与归因。

## 6. 工程与交付（保留核心）

### 6.1 技术栈

- 前端：Next.js + ECharts + ReactFlow + Tailwind。
- 后端：FastAPI + 多Agent编排 + LiteLLM网关。
- 数据：PostgreSQL/pgvector + Neo4j + Redis + TimescaleDB。
- 部署：Docker Compose（POC）+ Helm（生产）。

### 6.2 关键目录（简表）

- `frontend/`：企业版六页面（cockpit/device/optimize/catalyst/evolution/agent-ops）。
- `backend/api/`：场景API与路由注册。
- `backend/engine/`：在线演进、离线进化、安全门。
- `backend/industries/`：行业场景包。
- `data_pipeline/`：抓取、治理、数据集与种子数据。
- `deploy/`：Compose与Helm。
- `tests/`：unit/integration/e2e。

## 9. 质量门禁与自我迭代机制

### 9.1 约束力分级

`Runtime Gate >= CI/Tests >= Rule >= Doc`

### 9.2 必做门禁

- 关键目录变更（`data_pipeline/`、`backend/engine/`、`backend/industries/`）必须同步变更 `tests/`。
- PR必填：数据治理影响声明 + 变更到测试映射。
- 回归命令：`PYTHONPATH=. pytest tests/unit tests/integration tests/e2e -v`

### 9.3 每轮自检五问

1. 本轮是否新增不可追溯数据路径？
2. 质量分数是否真实影响策略准入（而非仅记录）？
3. 去重/增量是否误杀关键数据？
4. 是否有“测试通过但业务风险上升”的假阳性？
5. 本轮新增的人工检查项，何时补成自动化？

### 9.4 迭代节奏

- 每阶段进行一次红线评审（不过线不进下一阶段）。
- 每两轮执行机制审计（检查规则是否被绕过）。
- 形成“问题-改进-验证”三联记录，持续压降风险。
