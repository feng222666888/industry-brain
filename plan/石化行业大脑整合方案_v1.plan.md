# 石化行业大脑产品研发方案

## 0. 产品整体定位与展示逻辑（基于行业大脑三层架构）

本产品旨在通过一个完整、连贯的“行业大脑主控制台（Web端）”，系统性地呈现“**核心运行 + 底座支撑 + 自我进化**”的三层架构。针对政府与头部企业，产品不是割裂地演示三个孤立场景，而是**将这三个场景作为“行业大脑”不同层级能力的具象化切面**，让客户感受到大脑是一个统一的智能底座，在同时支撑多个业务线。

- **控制台首页（宏观视角）**：展示整个石化工厂的数字孪生看板，包括装置状态、实时报警（接入场景三：设备预警）、实时工艺寻优收益（接入场景一），以及底层大模型/Agent的运行负载。
- **架构能力切片1：底座支撑系统演示（通过场景二切入）**：展示“知识中心”如何将抓取的海量文献、电镜图片转化为“石化行业知识图谱”，并为多模态大模型提供检索支撑。
- **架构能力切片2：核心运行系统演示（通过场景三切入）**：展示“Multi-Agent引擎”在处理设备故障时，如何通过“记忆系统”保持上下文，并调度多个Agent（监测/诊断/维修）协同工作。
- **架构能力切片3：自我进化引擎演示（通过场景一切入）**：全景展示大脑最核心的“在线演进+离线进化”双轮驱动，如何实现实时信号感知、策略匹配，并最终沉淀为高收率公式资产。

## 1. 核心演示场景映射与真实性保障

为了确保产品体现**专业性与真实性**（非玩具级别），以下是三大支撑场景与架构的严格映射及数据方案：

### 场景一：炼化工艺参数在线寻优（重点展示“自我进化引擎”）

- **系统架构映射**：
  - **核心运行层**：使用 Agent 系统调度，分配一个“寻优优化 Agent”进行任务处理，结合**记忆系统**管理当前工况上下文。
  - **自我进化引擎层**：完全遵循双轮驱动架构。
    - **在线演进**：通过 Auto-Log 感知实时信号（仿真），从知识中心匹配当前 SOP，通过显式门控（安全执行），最终沉淀优化规则。
    - **离线进化**：挖掘历史仿真时序数据，通过遗传算法/基因变异生成新策略，在沙盒引擎中进行高保真仿真验证，固化为高收率策略。
  - **底座支撑层**：调用工具库（优化算法库）和知识中心（反应动力学业务逻辑）。
- **决策依据**：炼油与基础化工存在机理复杂、多参数耦合的问题。参数寻优直接关系到收率与能耗，是头部企业“降本增效”最核心的诉求（对应清单中协同制造★★★场景）。
- **数据真实性保障（如何避免玩具化）**：
  - **知识与规则**：通过爬虫抓取知网、Google Scholar等学术库中关于FCC（催化裂化）等核心装置的机理模型参数，以及公开的《石化行业操作规程》。
  - **时序数据**：真实工厂的高频传感数据属于机密，无法爬取。我们将采用化工界权威的**开源工业仿真数据集**（如 Tennessee Eastman Process, TEP），结合抓取的真实机理参数，生成带有真实工况波动的连续时序数据。

### 场景二：催化剂电镜图像智能识别（重点展示“多模态与知识中心”）

- **系统架构映射**：
  - **底座支撑层**：重度依赖**知识中心**中的“行业知识图谱”，以及**工具库**中的视觉大模型 API。
  - **核心运行层**：通过 Agent 构建系统创建一个“材料研发 Agent”，利用**可观测系统**记录其分析图像、推理结论的全过程。
- **决策依据**：传统研发痛点在于特征量化与性能关联难。这代表了“新质生产力”的研发新模式（对应清单协同研发★★场景），对政府客户展示极具科技前瞻性。
- **数据真实性保障（如何避免玩具化）**：
  - **图像与指标**：定向爬取开源材料科学顶级期刊（如 MDPI, Nature Comm）以及 Google Patents 中的专利。提取真实的 SEM/TEM 高清电镜图片，及孔径、比表面积等真实实验指标数据。
  - **知识图谱**：利用大模型从上述真实文献中抽取实体关系，构建完全基于真实科研成果的催化剂知识图谱。

### 场景三：关键设备预测性维护（重点展示“Multi-Agent 协同与工具调用”）

- **系统架构映射**：
  - **核心运行层**：重度展示 **Multi-Agent 引擎**。分配“监测 Agent”（负责时序异常检测）、“诊断 Agent”（负责调用外部知识库定损）、“维修 Agent”（生成维修 SOP）。展示 Manager/Handoffs 的 Agent 协同模式。
  - **自我进化引擎层**：展示**在线演进**中的“资产沉淀”（将成功的维修案例固化为优化版 SOP）与“协同共享”（同步到企业知识库）。
  - **底座支撑层**：利用工具库（知识检索功能）查询官方手册。
- **决策依据**：大型连续性生产对非计划停车极度敏感。通过时序大模型预测设备故障，直击管理层最关心的安全与效益生命线。
- **数据真实性保障（如何避免玩具化）**：
  - **知识与SOP**：爬取国际知名压缩机/泵阀厂商（如 Atlas Copco）的公开产品手册、维修指南，解析为结构化维修SOP存入知识中心。
  - **时序数据**：接入国际公认的权威机械故障工业数据集（如美国凯斯西储大学 CWRU 轴承数据集），极具说服力。

## 2. 技术栈决策与依据

为了支撑多模态AI、多智能体协同以及高品质的演示UI，采用以下技术架构：

### 前端展示层：React (Next.js) + ECharts + ReactFlow

- **依据**：演示产品需要极佳的UI交互。Next.js 提供优秀的加载性能；ECharts 用于处理海量传感器时序数据的可视化；**ReactFlow** 是展现“多智能体工作流”、“在线/离线进化链路”及“知识图谱节点”的最佳工具，优于传统的 Vue 生态。

### 后端与AI编排层：Python (FastAPI) + LangChain/AutoGen + 本地大模型生态

- **依据**：产品深度依赖爬虫（Scrapy/Playwright）、数据处理（Pandas）和 AI大模型调用。Python是唯一原生适配AI生态的语言，使用 FastAPI 可以提供高性能的异步接口，极大地降低系统复杂度（无需多语言间进行RPC通信）。
- **私有化部署适配**：考虑到政府与头部企业内网隔离的要求，AI底层调用不能硬编码依赖 OpenAI 等公有云 API。架构上引入 **LiteLLM** 作为统一网关，向下兼容并一键切换到私有化部署的本地开源大模型（如 Qwen-Max, DeepSeek-R1），使用 **Ollama / vLLM** 提供本地推理服务。

### 存储与知识中心层：PostgreSQL (pgvector) + Neo4j + Redis

- **依据**：关系型业务数据存储在 PostgreSQL；利用 pgvector 插件原生存向量实现 RAG（检索增强生成）；Neo4j 作为图数据库，用于构建底层的“行业知识图谱”（如：催化剂-反应-产物关系），为智能体提供事实依据。

### 私有化容器部署架构：Docker Compose / Kubernetes (K8s)

- **依据**：为了保证我们在开发环境抓取和治理好的数据能够平滑迁移至客户的断网环境，整个应用体系必须容器化。提供开箱即用的 Docker Compose 编排文件用于单机 POC 演示，并提供 K8s Helm Charts 支持未来在客户数据中心的集群化高可用部署。

## 3. 爬虫与数据治理链路设计

针对您确认的“全覆盖”数据源，设计如下数据流水线。**请注意，为了适应私有化部署，此流水线主要在我们本地/开发环境离线运行，其产出将打包为种子数据（Seed Data）随产品一并交付客户。**

1. **离线数据抓取 (Scraping)**：利用 Python 爬虫定时定点抓取：
  - *政府与协会*（如国家统计局）：宏观政策与环保安全合规标准。
  - *学术与专利库*（如知网）：催化剂配方、聚合反应机理。
  - *大型设备官网*：泵、压缩机等关键设备的官方维修手册与故障代码表。
2. **清洗与抽取 (Governance)**：
  - 将多源异构数据（PDF手册、网页文本）转换为 Markdown。
  - 利用大模型进行**信息抽取 (NER)**，提炼实体（Entity）与关系（Relation）。
3. **数据打包与私有化分发 (Package & Deploy)**：
  - 结构化关联数据写入 Neo4j 图数据库，形成“知识中心”；文本切块嵌入后存入 pgvector。
  - 核心步骤：将上述治理好的数据库导出为 Docker 初始化挂载脚本（init.sql, Neo4j dumps）。当产品在客户内网私有化启动时，自带完备的行业知识库，实现“开箱即用”。
