# 三大场景数据真实性与质量评估（当前版本）

## 1) 当前数据来源状态

- 场景3（设备预测性维护）：
  - 使用 CWRU 轴承公开数据作为参考来源（配置说明在 `data_pipeline/datasets/petrochemical/cwru/README.md`）。
  - 当前仓库中默认产物为 **CWRU 风格的合成数据**（用于演示和开发），由 `preprocess.py` 生成。
- 场景1（工艺寻优）：
  - 采用离线进化仿真与参数评分模型，当前属于可解释规则 + 仿真数据，不是企业生产实测全量数据。
- 场景2（催化剂研发）：
  - 当前为 MVP：知识图谱节点、催化剂库与文献引用为内置样例数据，尚未落地自动联网抓取管道。

## 2) 当前数据量（仓库内可复现）

- 场景3 CWRU 风格数据：
  - 4 类故障（normal / inner_race_fault / outer_race_fault / ball_fault）
  - 每类 50 窗口，共 200 窗口
  - 生命周期序列 720 小时（30 天）
- 场景1 进化数据：
  - 默认生成 15 代策略时间线，单代 population 为 10
- 场景2 催化剂数据：
  - 内置催化剂样例 3 条
  - 文献样例 3 条
  - 图谱节点 8、边 7

## 3) 是否“够用”

- 对“演示版/POC”够用：能完整跑通三大场景与自我进化主链路。
- 对“生产决策”不够：需要接入企业实测数据、扩大样本覆盖、增加跨装置与季节工况。

## 4) 质量现状与风险

- 已具备：基础字段完整性、参数范围、安全门拦截逻辑。
- 主要风险：
  - 合成数据与真实工况存在分布偏差；
  - 场景2知识图谱尚未自动化更新；
  - 抓取合规（robots/copyright）需要更严格策略（当前是基线检查）。

## 5) 自检、反思、自我迭代机制

每次数据变更执行以下闭环：

1. **真实性审计**：标注来源 URL、抓取时间、采集方法、可追溯 ID。
2. **质量审计**：运行 `quality` 校验（完整性 + 物理范围 + 异常比例）。
3. **语义审计**：运行 `semantic` 术语标准化，检查行业术语冲突。
4. **合规审计**：运行 `compliance` 基线，阻断不合法来源。
5. **演化联动审计**：确认进化引擎安全门对越界策略有拦截与日志。
6. **回归测试**：执行 `PYTHONPATH=. pytest tests/unit tests/integration tests/e2e -v`。

## 6) 与自我进化系统的对应关系

- 在线进化：策略执行前强制经过安全门（防止异常参数进入执行）。
- 离线进化：仿真评分对偏离工艺最优区间的策略进行惩罚。
- 建议下一步：
  - 将数据质量评分作为策略准入权重输入（低质量数据降权）；
  - 增加数据漂移检测，触发“重训练/重演化”任务；
  - 将数据治理审计结果写入资产沉淀表，形成可追溯链路。

## 7) 约束力分级（深度反思结论）

仅依赖 Doc 的约束力 **不够**。建议执行以下分级：

1. **Doc（解释层）**：说明原则、背景、风险与方法论。
2. **Rule（行为层）**：约束 AI/开发过程中的必做动作（例如变更-测试映射、治理影响说明）。
3. **CI + Tests（门禁层）**：强制验证，不通过即失败，避免“口头合规”。
4. **Runtime Gate（运行层）**：安全门与参数边界在执行时拦截风险策略。

执行优先级：`Runtime Gate >= CI/Tests >= Rule >= Doc`。

## 8) 自我迭代机制（本次升级后）

每轮迭代按以下顺序自检并反思：

1. 本次变更是否改变了数据真实性假设？
2. 是否新增了可被误用的数据入口（需补 compliance/quality 检查）？
3. 是否影响演化策略准入逻辑（需补 safety/evolution 回归）？
4. 是否已在 Rule 中固化新的过程约束？
5. 是否有“暂时人工检查”的空白项（必须登记补自动化计划）？
